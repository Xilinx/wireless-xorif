Set when DL data has been received when the sequence table is still updating.

int xocp_set_schedule ( uint16_t  instance,
uint8_t  mode,
uint8_t  length,
const uint8_t  sequence[] 
)

Set the "next" symbol processing schedule.

Parameters
instanceDevice instance ID (0..N)
modeMode (1 = DL, 2 = UL, 3 = both DL & UL)
lengthLength of sequence
sequencePointer to array defining the symbol sequence
Returns
  • XOCP_SUCCESS on success
  • Error code on failure
Note
The HW supports separate schedules for DL and UL. This API call can be used to program them separately (mode 1 or 2) or together (mode 3). The "sequence" parameter is an array (pointer) defining the sequence of component carrier symbols to process, e.g. {0, 0, 1} means a symbol from CC0, another symbol from CC0, and a symbol from CC1. Note, the component carrier IDs used here are the O-RAN CC IDs. Note, the actual symbol sequence needs to be coordinated with the DFE processing chain, and take into account the availability of symbols to/from the Front Haul Interface. This is left to the application SW.

int xocp_start ( const char *  dev_name)

Start a driver instance for the device.

Parameters
dev_nameDevice name of the O-RAN Channel Processor (or NULL)
Returns
  • Instance number (0..N)
  • Error code on failure
Note
If "dev_name" is left as NULL, then the library will search the devices for the most appropriate one. Alternatively, "dev_name" can be used as hint, e.g. "device_xyz" will match "a00000000.device_xyz". Specify the exact device name if you want to override this behavior.
